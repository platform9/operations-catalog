Operations Catalog — API Reference

Service: Operations Catalog API  ·  Base URL: http://localhost:5000  ·  Framework: Flask + psycopg2  ·  Database: PostgreSQL

1. Endpoint map

All available routes, HTTP methods, and a short description of each operation.

Method Route Description Response
GET /healthz Database connectivity check 200 {status}
GET /catalog Returns all catalog entries ordered by id 200 [ ]
POST /catalog Creates a new entry — serviceName required 201 { }
GET /catalog/:id Returns a single entry by numeric id 200 { }
PUT /catalog/:id Full update of an entry by numeric id 200 { }
DELETE /catalog/:id Deletes an entry by numeric id 200 {deleted}
GET /catalog/name/:name Returns a single entry by serviceName 200 { }
PUT /catalog/name/:name Full update of an entry by serviceName 200 { }
DELETE /catalog/name/:name Deletes an entry by serviceName 200 {deleted}
⚠ /catalog/name/:name routes are case-sensitive. Use LOWER("serviceName") = LOWER(%s) in the query to make lookups case-insensitive.

2. Request / response flow

How a request travels from the client through Flask middleware and into Postgres, then returns as JSON.

Client
index.html — browser fetch() HTTP request with JSON body (where applicable)
Middleware
before_request — init_db() Creates the catalog table if it does not exist
Flask
Route handler Matched by method + path — GET / POST / PUT / DELETE
DB layer
get_db() via Flask g Per-request psycopg2 connection — reused within the same request
Postgres
catalog table SQL executed — rows returned or written
Serialize
deserialize(row_to_dict()) JSONB string fields parsed back into Python dicts / lists
Teardown
teardown_appcontext — close_db() Connection closed after each request completes
Response
JSON response to client 200 / 201 with entry data, or 400 / 404 with error detail

3. Catalog table schema

PostgreSQL table definition. JSONB fields are stored as structured JSON and deserialized on read.

Column Type Notes
id SERIAL PRIMARY KEY Auto-incrementing integer
serviceName TEXT NOT NULL Unique name of the service
health JSONB JSONB Array of {key, value} pairs — e.g. overall, inMaintenance
description TEXT Human-readable description of the service
status TEXT e.g. Active, TBD
serviceCategory TEXT e.g. Operated Services, Supported Services
serviceSubjectMatterExperts JSONB JSONB String array of team or person names
criticalDependencies JSONB JSONB String array of dependency names
documentation JSONB JSONB String array of documentation URLs
SLA JSONB JSONB Object with availability, responseTimes, resolutionTargets
targetAudience TEXT
requestsChannel TEXT
incidentManagement TEXT
monitoringTools TEXT
activeMaintenanceWindows TEXT
onboardingDocumentation TEXT
costModel TEXT
versionInformation TEXT
deprecationPolicy TEXT

4. PUT operation flow

Step-by-step data flow when updating an existing catalog entry. Applies to both /catalog/:id and /catalog/name/:name.

Client
PUT request with JSON body Only the fields to update need to be included
Flask
Route matched PUT /catalog/:id or PUT /catalog/name/:name
DB read
SELECT existing row WHERE id = %s or WHERE "serviceName" = %s
✓ Found
Proceeds to merge step
✗ Not found
Returns 404 with error detail
Merge
current.update(data) Incoming fields overwrite existing values — unset fields are preserved
Serialize
json.dumps() on JSONB fields SLA, documentation, criticalDependencies, serviceSubjectMatterExperts
DB write
UPDATE catalog SET … WHERE id = %s All columns written — conn.commit()
Response
200 — full updated entry returned as JSON Re-fetched from DB after commit to reflect the saved state
